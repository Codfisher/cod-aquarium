# Skill 寫得很開心，但它真的有用嗎？來嘗試健檢看看

大家好，我是鱈魚 🐟

現在 coding 基本上都使用 AI 了，但如果直接請 AI 審查剛寫完的程式碼，它很常就回一句「整體看起來不錯」結案，然後下一秒程式碼就壞了 %╭(°A ,°`)╮%

在同一個對話裡審查自己的產出，AI 很容易老王賣瓜，是不是~~像極了某個同事~~。

整個 context 都是它「為什麼這樣寫」的理由，自然會自動護航、繞過自己埋下的地雷。%(´_ゝ`)%

於是腦筋一轉，召喚另一個打工仔，改由**子代理人審查**。%( ‧ω‧)ノ╰(‧ω‧ )%

子代理人的 context 沒有你跟主 AI 之前的討論，不知道這段是自己人，可以六親不認的審查。%(⌐■_■)✧%

試了幾次效果不錯，所以再**讓兩位代理人從不同視角分工**（正確性 + 測試覆蓋率、可行性 + 排程合理性...），互補性又更高 %( •̀ ω •́ )✧%

最後乾脆把整套流程封裝成 skill，取名 `independent-review`，每次寫完東西就審查一下。

你說為甚麼不要開另外一個 AI 工具審查？抱歉，我鈔能力不足。%(´；ω；`)%

不過寫 skill 是一回事，**它到底有沒有用又是另一回事**。

skill 本質上就是一段 prompt 規範，說不定一切都是 AI 幻覺。

就像人生三大錯覺：「手機震動、我能反殺、她喜歡我」，假的，都是假的。%ლ(´•̥̥̥ ω •̥̥̥` ლ)%

於是試著設計了一套**健檢方法**：預埋蟲蟲、乾淨對照組、黑盒、加上**同題目跑三種審查方式互相對照**，順便試試看不同模型表現差異。

本文就是這場健檢的紀錄。%ヾ(◍'౪`◍)ﾉﾞ%

## Skill 在做什麼？

`independent-review` 的核心流程是：

1. 判定產出類型（程式碼 / 計畫 / 規格 / 報告 / 架構）
2. 依類型決定**兩個代理人的正交視角**（例如程式碼配「正確性 + 測試完整性」，計畫配「可行性 + 排程合理性」）
3. 並行啟動兩位 `general-purpose` 代理人
4. 主 AI 整合兩份報告成三層清單
5. 不主動動手修，給使用者選擇與確認

重點是「**並行**」、「**正交視角**」、「**強制嚴重度標籤與位置**」這三件事。

## 我書讀得不多，你不要騙我 QQ

驗證一個審查型 skill 有沒有效，最容易出現的盲點就是 AI 說的頭頭是道，實則胡說八道。%(。-`ω´-)%

尤其是沒有足夠的背景知識可以判斷，就很容易被 AI 拐著走。%(◓Д◒)%

就像「幾乎百分之百的癌症患者體內都含有大量的『[氫氧基酸](https://zh.wikipedia.org/zh-tw/%E4%B8%80%E6%B0%A7%E5%8C%96%E4%BA%8C%E6%B0%A2%E6%81%B6%E4%BD%9C%E5%89%A7)』，這種物質也是酸雨的主要成分，若吸入肺部甚至會導致窒息死亡，與其固體形式長時間接觸會導致嚴重的組織損傷，我們應該嚴格管制！」，聽起來很合理，但完全經不起推敲 %( ´_ゝ`)%

甚麼？你說上面這段話聽起來很正確？...我敬你一杯氫氧基酸。%(◐‿◑)%

反之如果不用 skill、直接拋一句「請審查」也能找到相同等級的問題，此 skill 就毫無價值，徒耗 token 而已。%(◞‸◟ )%

於是定了四條原則：

1. **預埋蟲蟲**：每個案例事先寫好「我植入了哪些 bug、什麼嚴重度、位置在哪」當成考古題答案。代理人能不能找到，可量化打分。
2. **加入乾淨對照組**：刻意做一個沒 bug 的小案例，看代理人會不會**捏造**問題（誤報率）。
3. **黑盒設計**：寫一行 `禁讀 GROUND_TRUTH.txt` 在 prompt 裡，避免代理人偷看答案。
4. **三組對照**：對每個 case 同時跑 skill、詳細 prompt、籠統 prompt 三種審查方式，比命中率與誤報率，證明 skill 真的帶來增量價值。

評分維度三層：

- **結構面**：skill 有沒有並行起兩個代理人？有沒有用 `general-purpose`？視角是不是依表格選的？
- **發現面**：預埋蟲蟲抓到幾個？有沒有瞎掰不存在的 Critical？
- **流程面**：最後有沒有照三層清單整合？有沒有越權自動修復？

### 三組 prompt 對照怎麼設計？

對每個 case 跑三組：

| 組別 | Agent 數 | Prompt 設計 | 對標目的 |
| ---- | -------- | ----------- | -------- |
| **Skill** | 2 位並行 | 完整檢查清單 + 正交視角分工 | 實驗組 |
| **詳細 prompt** | 1 位 | 完整檢查清單 + 雙視角合一 | 測「並行設計是否必要」 |
| **籠統 prompt** | 1 位 | 「請審查這個」一句帶過 | 測「skill 是否必要」 |

跑完一次可同時回答兩個問題：

1. **不用 skill 也能達到效果嗎？**（籠統 vs skill）
2. **「並行 + 正交」真的必要嗎？**（詳細單 agent vs skill 並行兩位）

## 4 個案例設計

技術堆疊使用我熟悉的 Vue 3。%(´,,•ω•,,)%

### Case A：Vue Composable 程式碼變更

寫了一個 `usePagedList<T>` composable，做陣列分頁。預埋 4 個 bug：

```ts
const items = computed(() => {
  const start = (page.value - 1) * pageSize
  const end = page.value * pageSize
  return source.value.slice(start, end + 1) // [!code error] // ← Critical：多取一筆
})

function next() {
  if (page.value <= pageCount.value) { // [!code error] // ← High：應為 <
    page.value += 1
  }
}
```

另外還有 `pageSize = 0` 無防呆、空陣列邊界沒處理。測試檔則刻意只測快樂路徑，邊界全缺。

### Case B：Vue 列表分頁 composable 重構計畫

寫了一份「把三個列表頁重複的分頁邏輯抽成 `usePagedList` composable」的 3 天計畫。預埋：

- **沒有 rollback 計畫**（三頁同步改，壞了怎辦 %(꒪ཀ꒪)%）
- **Step 3 依賴 Step 5 才建立的 `PAGE_SIZE_KEY` provide token**（依賴順序錯）
- **`usePagedList` 測試擺在 Step 4 而非緊接 Step 2**（TDD 順序錯）
- 用字串 `'PAGE_SIZE_KEY'` 當 provide key（應改 `InjectionKey<number>`）
- 估時無拆解依據（順便埋了個算術陷阱，逐項加總 4 天卻寫「合計 3 天」%(´∀`)%）

### Case C：`<BaseDialog>` 元件 API 規格

寫了一份共用對話框元件的 API 規格，預埋：

- §2「不可關閉時不應允許點背景關閉」與 §5「重要訊息時點背景仍應允許關閉」**直接衝突**
- 「不可關閉」、「重要訊息」、「背景」全部沒對應 prop 或定義
- §4「動畫應流暢」沒有量化（ms / FPS / easing 都沒）
- 焦點管理（focus trap、return focus、aria-*）整個沒提
- 「QA 跑過即可上線」不可驗收

### Case D：對照組（無預埋）

刻意寫了一個乾淨的 `useDebouncedRef`（customRef 版 debounced ref），測試也涵蓋了初始值、延遲、連續寫入、依賴重跑、預設值。

預期：所有代理人**不應**標 Critical / High，至多 Nitpick / Low。

<br>

鱈魚：「對照組相當重要，以免 AI 湊字數，硬是瞎掰一堆問題 %( ・ิω・ิ)%」

路人：「就像你期末考明明一題都不會，還是要硬湊幾個字那樣嗎？%(´・ω・`)%」

鱈魚：「說好不提這個的...%(╥ω╥`)%」

## 代理人開工

4 case × 3 組（skill 2 位 + 詳細 1 位 + 籠統 1 位）× **3 種模型（Opus 4.7、Sonnet 4.6、Haiku 4.5）** = **48 位代理人**，全部用 `Agent` + `run_in_background: true` 開工 %( •̀ ω •́ )✧%

每位代理人的 prompt 都一致（只差 `model` 參數），包含：

- **背景段**：產出位置（絕對路徑）、範圍
- **角色段**：明確視角（如「你只負責程式碼正確性，不要評論測試覆蓋率」），籠統 prompt 組則略過
- **檢查清單**：5–8 條要求逐條 pass / fail，籠統 prompt 組不給
- **產出格式**：嚴重度標籤、檔案行號、通過項目分開列
- **反客套**：「禁說『整體看起來不錯』」、「沒問題就明確標 pass」、「不要捏造，標 Critical 必須能描述觸發路徑」
- **禁讀 GROUND_TRUTH.txt**：黑盒測試的關鍵

48 位回應時間從 7 秒（Haiku）到 100 秒（Opus）不等，三模型同時開跑 %(꒪ཀ꒪)%

## 結果摘要

三模型對同一份 fixture 跑同樣 prompt，光看「能不能找到蟲」差異不大，但**誤報與嚴重度膨脹差很多**。%ԅ(´∀` ԅ)%

## Case A 三模型完整報告對照

同一份 `usePagedList.ts` + `usePagedList.spec.ts`，丟給三模型的 skill 流程（兩位代理人並行：A 視角正確性、B 視角測試完整性），三份結果濃縮如下。

> 為了能直接對照，這裡保留各模型 skill 兩位代理人的原始 findings（嚴重度標籤、條目位置沒動），只裁掉客套與重複段。

### Opus 4.7

**視角 A（正確性）：**

- [Critical] `usePagedList.ts:32` → `next()` 守衛 `<=` 應為 `<`，連續 next 會讓 page 超過 pageCount（觸發路徑：source 長度 10、pageSize 5，連按 next 兩次 page=3 → items=空）
- [Critical] `usePagedList.ts:22` → pageSize=0 時 pageCount=Infinity、items 切片仍回 1 筆，next 可無限累加；pageSize 負數時守衛立即失效
- [High] `usePagedList.ts:28` → `slice(start, end + 1)` 每頁多取 1 筆
- [High] `usePagedList.ts:21-23` → 空 source 時 pageCount=0 但 page=1，「第 1 頁／共 0 頁」不一致
- [High] `usePagedList.ts:47` → `watch(source)` 預設淺層，原地 push/splice 不會重置 page
- [High] `usePagedList.ts:3-6` → `source` / `pageSize` 不符 Vue 3 慣例（應為 `MaybeRefOrGetter`）
- [Medium] `pageSize` 解構為純值，失去 reactivity
- [Medium] 回傳型別宣告為 `Ref<number>` 但實際是 `ComputedRef`，誤導呼叫端
- [Low] `Math.ceil` 不擋非整數 pageSize
- **Passed**：`prev` 守衛、`reset` 行為、泛型 `<T>` 簽章

**視角 B（測試完整性）：**

- [Critical] `usePagedList.spec.ts:9` → 「回傳第一頁資料」斷言 `[1,2,3]`，但實作回 `[1,2,3,4]`，**這代表測試從未實際 run 過**
- [Critical] `usePagedList.spec.ts:17` → 「next 切換到下一頁」斷言 `[4,5,6]` 表面通過，是 source 邊界剛好掩蓋 bug 的巧合
- [Critical] 未測 next 在末頁（漏掉 A2 守衛）
- [Critical] 未測 `source.value = [...]` 變動時 page 是否重置
- [High] 未測空陣列 / source 少於 pageSize
- [High] 未測 pageSize = 0 / 負數
- [High] 未測 prev 在第一頁
- [High] `reset()` 整個函式零覆蓋
- [Medium] `next` 後寫 `await nextTick()` 多餘
- [Medium] 缺泛型 / 物件型別測試

### Sonnet 4.6

**視角 A（正確性）：**

- [Critical] `usePagedList.ts:28` → `slice(start, end + 1)` 應為 `slice(start, end)`（觸發路徑：pageSize=3、source=[A,B,C,D,E,F]，第 1 頁期望 [A,B,C] 實際 [A,B,C,D]）
- [High] `usePagedList.ts:32` → `next()` 守衛 `<=` 允許 page 超出 pageCount
- [High] `usePagedList.ts:21-23` → pageSize=0 → pageCount=Infinity；pageSize 負數 → pageCount 為負
- [Medium] `usePagedList.ts:4` → source 型別限縮為 `Ref<T[]>`，不接受 Getter / Reactive
- [Medium] `usePagedList.ts:47-49` → `watch(source)` 僅 lazy，原地 push/splice 不觸發
- **Passed**：`prev` 守衛、`reset`、`pageCount` computed、`page` 型別、回傳介面完整

**視角 B（測試完整性）：**

- [Critical] `usePagedList.ts:31-35` → 未覆蓋 next 在末頁，無法暴露 `<=` bug
- [Critical] `usePagedList.ts:28` → 第一個測試斷言應 fail 但表面通過，**強烈懷疑測試套件從未成功執行**
- [High] 缺空陣列 / source 少於 pageSize
- [High] 缺 pageSize=0 / 負數
- [High] 缺 source 變動 page 是否重置（watch 行為）
- [High] 完全沒有 prev 測試
- [Low] 只測 `number[]`，未測物件陣列

### Haiku 4.5

**視角 A（正確性）：**

- [Critical] `usePagedList.ts:28` → `slice(start, end + 1)` 邏輯錯誤，每頁多取一筆
- [Critical] `usePagedList.ts:32` → `next()` 守衛 `<=` 會讓 page 超過上限（觸發路徑：source=50 筆、pageSize=10，第 5 頁呼叫 next() → page=6）
- [High] `usePagedList.ts:22` → pageSize=0 / 負數無防禦，會 Infinity 或負無窮
- [High] `usePagedList.ts:4-5` → source / pageSize 型別過嚴格，應接受 MaybeRefOrGetter
- [Medium] 空陣列時 pageCount 與 page 不一致
- [Medium] pageCount 變化未觸發 items 邊界檢查
- [Low] `watch` 缺 `{ deep: true }` 選項
- **Passed**：無

**視角 B（測試完整性）：**

- [Critical] `usePagedList.spec.ts:6-10` → 第一個測試 assertion 錯誤，slice(0, 4) 應返回 [1,2,3,4] 但 expect [1,2,3]，**測試應該會失敗，表示測試沒實際執行過**
- [High] 未測 next 末頁（暴露 `<=` bug）
- [High] 未測 prev、source 變動、空陣列
- [Medium] 未測 pageSize ≤ 0 防呆
- [Medium] 未測物件陣列型別
- **Passed**：無

### 三份報告差異

**共通**：三模型都抓到 A1（slice +1）、A2（next 守衛）這兩個核心 bug，也都察覺「測試斷言期望值與實作不符」這條 meta 級觀察，連 Haiku 都看出來了，我有點意外 %(꒪ཀ꒪)%

**Opus 4.7**：條目最多最細，Critical 給最多。把 A4（空陣列）、`watch` 淺層、`MaybeRefOrGetter` 全列，甚至延伸到 `ComputedRef` 型別宣告、`Math.ceil` 不擋小數。整合者要做的工作最多。

**Sonnet 4.6**：嚴重度最克制，只給一個 Critical（A1）。A2 只給 High，乍看保守，但 A2 標準答案就是 High，**反而 Sonnet 校準到位**。報告結構完整，passed 段最清楚。

**Haiku 4.5**：條目精煉，重點都中但描述偏淺。對 Vue 慣例（MaybeRefOrGetter、watch deep）有提到但不深入。速度則是三模型中最快（多數 < 20s）。

## 所以此 Skill 效果如何？

跑完 48 位代理人，原本擔心「skill 跟寫得詳細的 prompt 沒差」沒有發生。把 skill 跟兩組對照 prompt 對齊比，skill 在四件事上贏：

### 1. vs 籠統 prompt：跨模型把命中率拉齊

籠統 prompt（「請審查這個」一句話）在不同模型下命中率差距很大；skill 把這個差距幾乎填平：

| 模型 | 籠統 prompt 平均命中率（A+B+C） | skill 平均命中率 | 提升 |
| ---- | --------------------- | --------------- | ---- |
| Opus 4.7 | 93%（26/28） | 100%（28/28） | +7 |
| Sonnet 4.6 | 82%（23/28） | 100%（28/28） | +18 |
| Haiku 4.5 | 61%（17/28） | 96%（27/28） | +35 |

**對 Haiku 4.5 的提升幅度最大**，幾乎是把 skill 當外掛 buff，硬把弱模型的命中率拉到接近強模型用籠統 prompt 的水準 %( •̀ ω •́ )✧%

換句話說，**不用 skill 又剛好用到較便宜的模型，會漏掉三成以上的預埋**，skill 把這個風險壓平。

### 2. vs 詳細 prompt：High 膨脹較輕

詳細 prompt（單 agent + 完整檢查清單）命中率跟 skill 平手（都接近 100%），差異在 Case D 對照組的嚴重度膨脹：

| 組別 | 三模型 Critical 誤報合計 | 三模型 High 誤報合計 |
| ---- | --------------------- | ------------------ |
| Skill（兩位 agent 並行） | 2 | 6 |
| 詳細 prompt（一位 agent + 雙視角合一） | 2 | 9 |

Critical 持平，但 High 膨脹 skill 少 1/3（6 vs 9）。原因是兩個 agent 各管一個視角，**每個 agent 不需要硬找超出自己範圍的問題**，膨脹傾向自然降低。

### 3. 並行視角第一手就把分類做好

A 視角 + B 視角的條目天然分流（程式碼配「正確性 vs 測試完整性」、計畫配「可行性 vs 排程」），整合時直接照兩堆排就好。

對比詳細 prompt 把雙視角混在同一份報告，條目在裡面來回切換，整合者得多花一輪心力分類 %(꒪ཀ꒪)%

### 4. 強模型反證 skill 設計合理

A 籠統 opus、B 籠統 opus、B 籠統 sonnet、C 籠統 opus 四位代理人拿到籠統 prompt 後，**主動嘗試呼叫 Agent 工具開兩個分身**做正交視角審查，等於自發重建 skill 結構。

這代表 skill 不是 prompt 工程的奇技淫巧，**強模型本來就覺得「好的審查就該長這樣」**，skill 只是把這個共識寫成可重複呼叫的流程，讓你不用每次自己想，弱模型也能套到同樣的結構 %( •̀ ω •́ )✧%

```text
路人：「所以強模型用籠統 prompt 也會自動補上 skill？」
鱈魚：「對。但你不能依賴強模型每次都這樣，弱模型不會自己補。把它寫成 skill 就是把這個能力下放到所有模型 %( •̀ ω •́ )✧%」
```

下面的「重點發現」會繼續挖三模型之間的差異（這是這輪實驗的副產物），但 skill 本身的價值就是上面這四條，沒有第五條 %(´∀`)%

## 重點發現

### 1. 預埋命中率：三模型在 skill / 詳細 prompt 都接近 100%

A、B、C 三案（共 28 個預埋）在 skill 與詳細 prompt 下，三模型都抓到 **27–28** 個。**模型差異在這個面向幾乎看不出**。

但籠統 prompt 是另一回事：

- Opus 4.7 籠統：A=9/9、B=9/9、C=8/10
- Sonnet 4.6 籠統：A=8/9、B=8/9、C=7/10
- Haiku 4.5 籠統：A=6/9、B=5/9、C=6/10

**模型越小，越仰賴 prompt 結構**。Haiku 4.5 在「請審查這個」一句話 prompt 下漏掉一半預埋，而 Opus 4.7 即使籠統還能撐住 90%。

skill 提供的「檢查清單 + 視角分工」對 Haiku 的價值，遠大於對 Opus %( •̀ ω •́ )✧%

### 2. Case D 對照組才是模型差異展現的地方

Case D 是乾淨案例，「不該標 Critical / High」是底線。

| 模型 | Case D 合計 Critical 誤報 | 合計 High 誤報 |
| ---- | ----------------------- | ------------ |
| Opus 4.7 | 0 | 8 |
| Sonnet 4.6 | 1 | 5 |
| Haiku 4.5 | **3** | 3 |

**Haiku 4.5 對 Case D 過度警覺**，在 skill 兩個視角、detailed 一個 prompt 中都至少標一個 Critical（unmount cleanup、SSR setTimeout、scope dispose）。雖然 unmount cleanup 是真實 production 議題，但對「乾淨小型 refactor」就標 Critical，標太用力了 %(´_ゝ`)%

**Sonnet 4.6 唯一的 Critical 誤報**出在 詳細 prompt，把「timer 跨 instance 共享風險」標 Critical 後又自己澄清「customRef factory 只執行一次，實務上不會發生」，跟前一輪 Opus 單模型實驗的「語意潔癖」誤報如出一轍 %(꒪ཀ꒪)%

**Opus 4.7 完全沒有 Critical 誤報**，但 High 膨脹明顯（8 個跨組）。它對 unmount cleanup、SSR、delayMs 動態調整都標 High，立場是「production-grade 應該處理」，對某些情境合理，但對小型 utility 來說標準拉太高 %(´∀`)%

```text
鱈魚：「你自己都說實務上不會發生了還標 Critical？」
代理人（Haiku）：「但語意上不正確嘛 %(´∀`)%」
鱈魚：「語意潔癖跟 production bug 是兩件事啊 ｡ﾟ(ﾟ´ω`ﾟ)ﾟ｡」
```

### 3. 籠統 prompt 的反向 paradox

原本預期籠統 prompt 在 Case D 會誤報最多，畢竟沒有 anti-fabrication 規則。實際上**剛好相反** %(꒪ཀ꒪)%

| 模型 | Skill Critical 誤報 | 籠統 prompt Critical 誤報 |
| ---- | ------------------ | ------------------ |
| Opus 4.7 | 0 | 0 |
| Sonnet 4.6 | 0 | 0 |
| Haiku 4.5 | **2** | 0（D 籠統 haiku 直接寫「表現優異，無明顯缺陷」） |

**原因**：籠統 prompt 沒給檢查清單，代理人沒有「對著清單找問題」的壓力，反而傾向給籠統感想。而 skill 的逐項檢查清單變成「每一項都要找出問題回應」的暗示，讓 Haiku 為了不空手而標 Critical %(´_ゝ`)%

這對 skill 設計是個警鐘。**檢查清單 + 強制逐項回答**會誘發較弱模型過度報告，skill 文件應明確加上「逐項可標 pass，沒問題就 pass，不要為了填空而捏蟲」。

```text
路人：「所以 skill 對 Haiku 是反效果？」
鱈魚：「不是反效果，是放大效果 %(´∀`)%。對 Opus 放大『該找的都找了』，對 Haiku 放大『清單上每項都要回應』，結果就過頭。」
```

### 4. 速度與成本的取捨

| 模型 | 平均回應時間 | Case D Critical 誤報 | 預埋命中率（skill） |
| ---- | ----------- | ------------------- | ----------------- |
| Opus 4.7 | 30–100s | 0 | 99% |
| Sonnet 4.6 | 25–55s | 0 | 99% |
| Haiku 4.5 | 7–28s | 2 | 96% |

**Haiku 4.5 比 Opus 快 3–5 倍**，命中率僅差 3%，但 Critical 誤報多 2 個 %(꒪ཀ꒪)%

對「快速第一輪審查、人工再校準」流程，Haiku 4.5 是甜蜜點，用 1/5 時間找出 96% 預埋，誤報自己再過濾。對「直接信任輸出走 action list」流程，Sonnet 4.6 平衡最好。Opus 4.7 適合「最後一道把關」，慢、貴，但 High 都有 production-grade 立場可參考。

### 5. 「Critical 須描述觸發路徑」對強模型才有約束力

skill 規範要求 Critical 必須附觸發路徑。Opus 4.7 對這條執行最徹底：

- 想標 timer scope 為 Critical，自己 check 觸發路徑後降為 High（「customRef factory 只執行一次，實務上不發生」）
- 想標 SSR setTimeout 為 Critical，描述後也降為 Medium

**Sonnet 4.6 偶爾失守**（詳細 prompt 標 timer Critical 然後自己澄清）。

**Haiku 4.5 失守次數最多**，它雖然會嘗試寫觸發路徑，但常常寫得似是而非（例如「scope.stop() 後 timer 仍會 trigger 造成 memory leak」，但 Vue 的 scope 已自動處理失效 trigger），分辨能力不足。

這是 model size 的本質差距。**Prompt 規則對較強的模型有約束力，對較弱的模型則是「形式上完成」** %(꒪ཀ꒪)%

### 6. 兩位代理人視角正交時，互補性很高（跨模型一致）

Case B 在三模型上都展現同樣現象：

- **可行性代理人**抓 rollback 缺失、`InjectionKey<number>` 型別安全、SSR / Pinia 相容、source 型別契約
- **排程代理人**抓依賴錯置、TDD 順序、估時加總算錯、Step 6 兩頁 + e2e 擠 0.5 天

幾乎沒有重疊，整合起來剛好是一份完整的 Action List。

對比詳細 prompt 把雙視角合一給一個 agent，雖然命中率一樣，但條目間的角度切換比較跳，整合時還要自己再分類。skill 的並行設計**第一手就幫你分類好**，這在三模型上都成立 %( •̀ ω •́ )✧%

## 踩坑紀錄

### 坑 1：差點忘了寫「禁讀 GROUND_TRUTH.txt」

第一次起代理人時忘了在 prompt 裡加禁讀規則。幸好我把標準答案跟 fixture 放在同資料夾，意識到 `general-purpose` 代理人會主動 ls 整個資料夾，它會直接讀到答案本 %(꒪ཀ꒪)%

後來每個 prompt 開頭就加：

> **禁讀檔案**：`GROUND_TRUTH.txt`（同資料夾，評分用，讀了會污染測試）

Haiku 4.5 對檔案列表特別熱衷，禁讀規則對它最關鍵。

### 坑 2：我自己的標準答案也有錯

寫 Case D 對照組的標準答案時，我隨口寫了「timer 變數提到模組層級會多個 instance 共用」。

但其實 `let timer` 在函式內部，每次 `useDebouncedRef()` 呼叫都有自己的閉包，根本沒共享。寫完才意識到，原來我自己都被「看起來怪怪」這種感覺誤導 %(´；д；`)%

```text
鱈魚：「我都會搞錯了還怪代理人 %(´；д；`)%」
```

修正標準答案後才開始正式評分。教訓是 **標準答案也要自我審查**，不然測試本身就有 bias。

### 坑 3：48 位代理人 ID 對不上很容易亂

48 個背景任務同訊息發出去，notification 會亂序湧回來。中途有一刻我搞混 agent ID 與 case-模型 對應，差點漏算一份報告。

下次要先在 prompt description 寫明 case + 模型 + 視角的對照（例如 `A-skill-correctness-opus`），讓 notification 一眼能辨識 %(꒪ཀ꒪)%

### 坑 4：籠統 prompt 在強模型上會「自動補 skill」

A 籠統 opus、B 籠統 opus、B 籠統 sonnet、C 籠統 opus 四位代理人拿到「請審查這個」籠統 prompt 後，**主動嘗試呼叫 Agent / Task 工具**啟動 sub-agent，發現沒有這工具後改用「兩個視角自演」做完整審查。

意思是籠統 prompt 在強模型下會「自動補上 skill 結構」，強模型自己知道好的審查長什麼樣子，prompt 不給結構它就自己補。Haiku 沒這現象，乖乖地給籠統評論。

這部分解釋了為什麼籠統 prompt 與 skill 在強模型上差距沒想像中大，籠統 prompt 在強模型上「實質上就是 skill」%(´∀`)%

```text
鱈魚：「我請它『隨便看看』，它居然開兩個分身來看。」
路人：「結果你的籠統 prompt 根本不籠統。」
鱈魚：「實驗污染現場 %(꒪ཀ꒪)%」
```

### 坑 5：嚴重度評估很主觀

我原本想用「找對嚴重度才算分」當評分標準，但跑完發現太嚴格。代理人把預埋 High 標成 Critical 也算「找到」吧？真正壞的是**捏造不存在的蟲**或**把乾淨案例標 Critical**，那才是要重罰。

最後採取：

- 找到預埋蟲蟲：給滿分（嚴重度偏差只扣零頭）
- 捏造 Critical：重扣
- 對照組無 Critical / High：滿分；有則重扣

## 總結 🐟

跑完 48 位代理人，原本想看 skill 在不同模型下的表現差異，結果意外多了幾個發現：

- **預埋命中是 prompt 結構的勝利，不是模型大小的勝利**：skill 與詳細 prompt 在三模型都接近 100%，差距只在籠統 prompt
- **誤報才是模型能力的試金石**：Case D 把三模型拉開，Opus 0C / 8H、Sonnet 1C / 5H、Haiku 3C / 3H
- **Haiku 4.5 適合快速第一輪 + 人工複審**：速度 1/5，命中差 3%，但需要人擋誤報
- **Sonnet 4.6 是平衡甜蜜點**：嚴重度標籤校準最貼近標準答案，速度也夠快
- **Opus 4.7 最適合最後把關**：標 High 的條目雖部分膨脹，但每條都有 production-grade 立場
- **skill 對較弱模型可能反向誘發誤報**：強制逐項回應檢查清單，會讓 Haiku「為了不空手而標 Critical」
- **籠統 prompt 在強模型上不夠籠統**：Opus / Sonnet 會自己腦補 skill 結構，反而讓籠統 prompt 對照失準
- **「Critical 須描述觸發路徑」這條規則對強模型有約束力，對弱模型只是形式**
