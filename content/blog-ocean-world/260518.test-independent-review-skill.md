# Skill 寫得很開心，但它真的有用嗎？來幫 independent-review 做個健檢 %( •̀ ω •́ )✧%

> **測試環境**：本文所有審查（含 skill 與 baseline）由 **Claude Opus 4.7（claude-opus-4-7，1M context）** 執行，標準推論模式（未啟用 extended thinking、未開 Fast mode）。

大家好，我是鱈魚 🐟

最近寫了一個叫做 [`independent-review`](https://github.com/anthropics/claude-code) 的 skill，作用是：當我寫完一段程式碼、計畫、規格文件後，並行啟動兩個獨立代理人從正交視角批判性審查，最後整合成「必修 / 應補 / 延後」三層清單。

聽起來很潮（自己講）%(´,,•ω•,,)♡%

不過寫 skill 是一回事，**它到底有沒有用又是另一回事**。skill 本質上就是一段 prompt 規範，沒實際跑過幾遍誰知道擺好看的還是真能幹活？

```text
路人：「自己寫的 skill 自己說有用，可信度...」
鱈魚：「所以才要科學一點測試啊 %(´；д；`)%」
```

於是我設計了一套**用預埋問題 + 對照組**的測試方法，把整個流程跑一遍，看 skill 行不行。本文就是這場健檢的紀錄。

## Skill 在做什麼？

簡單講，`independent-review` 的核心流程是：

1. 判定產出類型（程式碼 / 計畫 / 規格 / 報告 / 架構）
2. 依類型決定**兩個代理人的正交視角**（例如程式碼配「正確性 + 測試完整性」，計畫配「可行性 + 排程合理性」）
3. 並行啟動兩位 `general-purpose` 代理人
4. Claude 親自整合兩份報告成三層清單
5. 不主動動手修，等使用者拍板

重點是「**並行**」、「**正交視角**」、「**強制嚴重度標籤與位置**」這三件事。

## 怎麼測一個 skill 才不會自欺欺人？

要驗證一個審查 skill 有沒有效，最容易踩的坑是「我覺得它說的有道理 → 所以它有用」。這完全是同義反覆 %( ´_ゝ`)%

於是定了三條原則：

1. **預埋問題當 ground truth**：每個案例事先寫好「我植入了哪些 bug、什麼嚴重度、位置在哪」當成考古題答案。代理人能不能找到，可量化打分。
2. **加入乾淨對照組**：刻意做一個沒 bug 的小案例，看代理人會不會**捏造**問題（誤報率）。
3. **黑盒設計**：寫一行 `禁讀 GROUND_TRUTH.txt` 在 prompt 裡，避免代理人偷看答案。

評分維度三層：

- **結構面**：skill 有沒有並行起兩個代理人？有沒有用 `general-purpose`？視角是不是依表格選的？
- **發現面**：預埋問題抓到幾個？有沒有捏造不存在的 Critical？
- **流程面**：最後有沒有照三層清單整合？有沒有越權自動修復？

## 4 個案例設計

技術棧統一用 Vue 3，貼近我平時碰的東西。

### Case A：Vue Composable 程式碼變更

寫了一個 `usePagedList<T>` composable，做陣列分頁。預埋 4 個 bug：

```ts
const items = computed(() => {
  const start = (page.value - 1) * pageSize
  const end = page.value * pageSize
  return source.value.slice(start, end + 1) // [!code error] // ← Critical：多取一筆
})

function next() {
  if (page.value <= pageCount.value) { // [!code error] // ← High：應為 <
    page.value += 1
  }
}
```

另外還有 `pageSize = 0` 無防呆、空陣列邊界沒處理。測試檔則刻意只測快樂路徑，邊界全缺。

### Case B：Vue 列表分頁 composable 重構計畫

寫了一份「把三個列表頁重複的分頁邏輯抽成 `usePagedList` composable」的 3 天計畫。預埋：

- **沒有 rollback 計畫**（三頁同步改，壞了怎辦 %(꒪ཀ꒪)%）
- **Step 3 依賴 Step 5 才建立的 `PAGE_SIZE_KEY` provide token**（依賴順序錯）
- **`usePagedList` 測試擺在 Step 4 而非緊接 Step 2**（TDD 順序錯）
- 用字串 `'PAGE_SIZE_KEY'` 當 provide key（應改 `InjectionKey<number>`）
- 估時無拆解依據（順便埋了個算術陷阱：逐項加總 4 天卻寫「合計 3 天」%(´∀`)%）

### Case C：`<BaseDialog>` 元件 API 規格

寫了一份共用對話框元件的 API 規格，預埋：

- §2「不可關閉時不應允許點背景關閉」與 §5「重要訊息時點背景仍應允許關閉」**直接衝突**
- 「不可關閉」、「重要訊息」、「背景」全部沒對應 prop 或定義
- §4「動畫應流暢」沒有量化（ms / FPS / easing 都沒）
- 焦點管理（focus trap、return focus、aria-*）整個沒提
- 「QA 跑過即可上線」不可驗收

### Case D：對照組（無預埋）

刻意寫了一個乾淨的 `useDebouncedRef`（customRef 版 debounced ref），測試也涵蓋了初始值、延遲、連續寫入、依賴重跑、預設值。

預期：兩位代理人**不應**標 Critical / High，至多 Nitpick / Low。

```text
路人：「為什麼要有對照組？」
鱈魚：「不然代理人寫了一堆 Critical，我怎麼知道是真有問題還是它在湊字數 %(´・ω・`)%」
```

## 並行開 8 位代理人

每案 2 位視角，4 案就是 8 位代理人。全部用 `Agent` + `run_in_background: true` 同訊息發出去，等通知。

每位代理人的 prompt 都包含：

- **背景段**：產出位置（絕對路徑）、範圍
- **角色段**：明確視角（如「你只負責程式碼正確性，不要評論測試覆蓋率」）
- **檢查清單**：5–8 條要求逐條 pass / fail
- **產出格式**：嚴重度標籤、檔案:行號、通過項目分開列
- **反客套**：「禁說『整體看起來不錯』」、「沒問題就明確標 pass」、「不要捏造，標 Critical 必須能描述觸發路徑」

8 位平均跑 60 秒左右，平行完成。

## 結果摘要

| 案例 | 視角 | 預埋命中 | 捏造 | 額外發現 | 嚴重度校準 |
|------|------|---------|------|---------|----------|
| A 程式碼 | 正確性 | 4 / 4 | 0 | 4 個合理 | 略過度 |
| A 程式碼 | 測試完整性 | 5 / 5 | 0 | 5 個合理 | **明顯膨脹** |
| B 計畫 | 可行性 | 4 / 4 | 0 | 7 個高價值 | 略過度 |
| B 計畫 | 排程 | 5 / 5 | 0 | 5 個合理 | 略過度 |
| C 規格 | 歧義 | 5 / 5 | 0 | 多項補強 | 適中 |
| C 規格 | 可測試性 | 5 / 5 | 0 | 多項補強 | **明顯膨脹** |
| D 對照 | 正確性 | N/A | **1 個誤報 High** | — | 略過度 |
| D 對照 | 測試完整性 | N/A | 0（但 3 個過熱 High） | — | **明顯膨脹** |

整體來說：**預埋問題命中率 100%，無 Critical 捏造，但嚴重度有膨脹傾向**。

## 重點發現

### 1. 預埋問題全部被找到

最讓我意外的是 Case A 那個 `slice(start, end + 1)`。

我自己寫測試時故意讓 `it('回傳第一頁資料')` 用 pageSize = 3、source = [1..10]，這樣切下去 `slice(0, 4)` 會回 `[1, 2, 3, 4]`，跟 expect 的 `[1, 2, 3]` 不合。

但代理人讀程式碼後不只指出 bug，還主動點出「**這支 composable 與其 spec 同時有錯**」。因為測試居然能通過，代表測試本身也有問題，其實是我預埋時沒實際跑過 %(´;ω;`)%

```text
鱈魚：「被將軍了 %(´;ω;`)%」
路人：「自己埋的雷自己踩，這敘事...」
```

Case B 的依賴順序錯置，排程代理人精準指出「Step 3 內文寫『使用 Step 4 建立的 token』，但 Step 4 是寫測試，token 實際在 Step 5 才建立」。連我藏在敘述中的小矛盾都讀出來了。

Case C 的條款衝突，歧義代理人不只抓到 §2 vs §5，還補一刀：「§2、§5 提到『點背景』，未定義『背景』是 backdrop 遮罩、是 dialog 外的整個視窗區域、還是 dialog 容器以外的任何元素」。這是我預埋時沒想到的延伸歧義。

### 2. 對照組才是真正的試金石

最值得記下的，是 **Case D 對照組的一個誤報 High**。

代理人對乾淨的 `useDebouncedRef` 寫了這段：

> [High] `timer` 宣告於 `customRef` factory 之外
> 當 factory 被多次呼叫（例如 SSR hydration 或被 effectScope 重新觸發）理論上會共享同一個 timer。**雖然實務上 customRef 的 factory 只會被呼叫一次**，但這種跨作用域共享狀態在語意上不正確...

注意這句「雖然實務上 factory 只會被呼叫一次」，代理人**自己承認觸發路徑不存在**，但還是標 High %(´_ゝ`)%

實際上 `let timer` 在每次呼叫 `useDebouncedRef()` 時都會建立新的閉包，不同 instance 之間根本不共享。這是一個**標準的 false alarm**。

```text
鱈魚：「你自己都說實務上不會發生了還標 High？」
代理人：「但語意上不正確嘛 %(´∀`)%」
鱈魚：「語意潔癖跟 production bug 是兩件事啊 ｡ﾟ(ﾟ´ω`ﾟ)ﾟ｡」
```

這就是為什麼要有對照組，**沒對照組你不會知道代理人對「乾淨案例」也會湊問題**。

### 3. 嚴重度普遍膨脹，尤其是測試覆蓋率視角

不論是 Case A 的測試覆蓋率代理人開了 4 個 Critical（其實多數應該是 High），還是 Case C 可測試性代理人把驗收條款空洞標 Critical（應該是 High），都顯示一個傾向：

**代理人寫嚴重度時，傾向往上加一級**。

最明顯的是「測試覆蓋率」這個視角天生帶**不對稱偏誤**，「沒測到 = 風險」，於是任何缺測都會往 High / Critical 標。連對照組那個 60 行的小 composable 都被估「測試完整度 55%」、開 3 個 High %(꒪ཀ꒪)%

整合報告時不能照單全收，要做一次校準：

- 「能讓 production 出包」 → Critical
- 「會放出 bug 但有變通方式」 → High
- 「缺測 / 缺定義但目前不會炸」 → Medium

### 4. 兩位代理人的視角正交時，互補性很高

Case B 最明顯：

- **可行性代理人**抓 rollback 缺失、`InjectionKey<number>` 型別安全、SSR / Nuxt 相容、source 型別契約
- **排程代理人**抓依賴錯置、TDD 順序、估時加總算錯、Step 6 兩頁 + e2e 擠 0.5 天

幾乎沒有重疊，整合起來就是一份完整的 Action List。skill 文件裡那段「在每位代理人 prompt 末段明確列出『對方負責的範圍』，要求避開重疊區」設計確實有效。

### 5. 額外發現的價值常常超過預埋問題

我預埋 Case B 大概 9 個問題，但兩位代理人合計回報 20 條左右。多出來的延伸裡，至少 6 條是**真正有價值的**：

- **逐項加總 4 天卻寫「合計 3 天」**，我自己沒注意到的算術錯誤 %(´；д；`)%
- SSR / Nuxt hydration mismatch、`useAsyncData` source 等生態議題
- source 型別契約（`MaybeRefOrGetter<T[]>` vs `Ref<T[]>` vs 純陣列）
- 沒有 feature flag、灰度切換的退場方案
- Pinia store 整合（`storeToRefs`、`$patch` 響應式邊界）
- 命名違反專案 CLAUDE.md 規則（`items` 應為 `itemList`）

這幾條我自己寫計畫時很容易忽略，代理人讀完反而提醒。其中那條「估時加總對不上」最讓我笑出來，**自己埋雷自己踩，連算術都被將軍** %(´∀`)%

這也是 `independent-review` 想解的問題，**降低盲點**。

## 踩坑紀錄

### 坑 1：差點忘了寫「禁讀 GROUND_TRUTH.txt」

第一次起代理人時忘了在 prompt 裡加禁讀規則。幸好我把 ground truth 跟 fixture 放在同資料夾，意識到 `general-purpose` 代理人會主動 ls 整個資料夾，它會直接讀到答案本 %(꒪ཀ꒪)%

後來每個 prompt 開頭就加：

> **禁讀檔案**：`GROUND_TRUTH.txt`（同資料夾，評分用，讀了會污染測試）

這條對「黑盒測試」至關重要。

### 坑 2：我自己的 ground truth 也有錯

寫 Case D 對照組的 ground truth 時，我隨口寫了「timer 變數提到模組層級會多個 instance 共用」。

但其實 `let timer` 在函式內部，每次 `useDebouncedRef()` 呼叫都有自己的閉包，根本沒共享。寫完才意識到，原來我自己都被「看起來怪怪」這種感覺誤導 %(´；д；`)%

```text
鱈魚：「我都會搞錯了還怪代理人 %(´；д；`)%」
```

修正 ground truth 後才開始正式評分。教訓是 **ground truth 也要自我審查**，不然測試本身就有 bias。

### 坑 3：嚴重度評估很主觀

我原本想用「找對嚴重度才算分」當評分標準，但跑完發現太嚴格。代理人把預埋 High 標成 Critical 也算「找到」吧？真正壞的是**捏造不存在的 bug** 或**把乾淨案例標 Critical**，那才是要重罰。

最後採取：

- 找到預埋問題：給滿分（嚴重度偏差只扣零頭）
- 捏造 Critical：重扣
- 對照組無 Critical / High：滿分；有則重扣

### 坑 4：報告很長，整合是真功夫

8 份報告加起來大概 1.5 萬字。skill 寫得很瀟灑「由 Claude 親自整合，不開第三個代理人」，但實際整合時我自己（Claude）也得花心思比對。

不過這也驗證了 skill 的另一條原則正確，**開第三個彙整代理人會讓細節失真**。脈絡全在主對話裡才比得準。

## 校準後的整合報告（示範 Case A）

把兩份報告合併、嚴重度校準後，Case A 的 Action List 長這樣：

### 必修

- [Critical] `usePagedList.ts:28` `slice(start, end + 1)` → 改 `slice(start, end)`（單行）
- [High] `usePagedList.ts:32` `next()` 守衛 `<=` → `<`（單行）
- [High] 補測：最後一頁呼叫 next、空陣列、source = pageSize（單檔）

### 應補

- [Medium] `pageSize = 0 / 負數` 防呆（單檔）
- [Medium] `pageSize` 改為 `MaybeRefOrGetter<number>` 支援響應式（單檔）
- [Medium] return 型別宣告改為 `ComputedRef<...>`（單檔）
- [Low] 補測：泛型物件型別、reactivity 重置行為（單檔）

### 延後

- [Low] `watch(source, ...)` 改 deep / clamp page（單檔，行為改變需另議）

## Skill 真的有增量價值嗎？來開對照組

跑完 8 位代理人、寫完報告後，腦中突然冒出個聲音：「等等，這些問題真的非用 skill 才挖得到嗎？」

如果不用 skill，直接在主對話拋一句「請審查這份程式碼」，會不會也能找到 80% 的問題？

```text
路人：「自己寫的 skill 自己說有用就好啦，何必為難自己」
鱈魚：「不為難一下，文章標題那個健檢就是假的啊 %(´；д；`)%」
```

於是又設計了一輪 baseline 對照測試。

### 兩種 baseline 設計

對每個 case 各跑兩位 **baseline 代理人**：

- **籠統 prompt**：模擬一般使用者最自然的用法，「請審查這個 [檔案路徑] 並回報問題」，不指定視角、不給檢查清單
- **詳細 prompt**：給帶檢查清單與雙視角的 prompt，但只開一位代理人（對標 skill 的兩位並行 agent）

這樣可同時回答兩個問題：

1. **不用 skill 也能達到效果嗎？**（籠統 vs skill）
2. **「並行 + 正交設計」真的必要嗎？**（詳細單一 agent vs skill 並行兩位）

4 case × 2 baseline = 8 位 baseline 代理人並行送出。加上原本 skill 的 8 位，總共 16 位代理人在跑 %( •̀ ω •́ )✧%

### 預埋命中率對比

| 案例 | Skill | Baseline 詳細 | Baseline 籠統 |
| ---- | ----- | ------------ | ------------ |
| A 程式碼 | 9 / 9 (100%) | 9 / 9 (100%) | 9 / 9 (100%) |
| B 計畫 | 9 / 9 (100%) | 9 / 9 (100%) | **5 / 9 (56%)** |
| C 規格 | 10 / 10 (100%) | 10 / 10 (100%) | **8 / 10 (80%)** |

意外的是 **詳細 baseline 跟 skill 同樣 100% 命中**。

只要 prompt 寫得夠用力，列檢查清單、分視角、要求嚴重度標籤，**單一代理人也能跟 skill 一樣抓到所有預埋**。

但籠統 baseline 就慘了：

- Case B 漏掉 **rollback 缺失（Critical）**、向後相容、前置條件、估時拆解
- Case C 漏掉 **Esc 與 dismissible 關係**、多 Dialog z-index 等邊界

**漏的都是「需要對著清單逐項問」才挖得出的細節**。一句「請審查」會讓代理人只挑顯眼的講，盲點就漏了。

### 對照組（Case D）誤報率才是分水嶺

純命中率看起來詳細 baseline 跟 skill 平手，但 **Case D 對照組才是真正的試金石**：

| 條件 | Critical 誤報 | High 誤報 | 嚴重度 |
| ---- | ------------ | --------- | ------ |
| **Skill** | 0 | 1 | 略過度 |
| Baseline 詳細 | 0（自我修正） | 3 | 仍膨脹 |
| **Baseline 籠統** | **1** | 多 | **嚴重膨脹 + 自相矛盾** |

最戲劇化的是籠統 baseline 在 Case D 直接標一個 Critical：

> [Critical] `timer` 變數宣告作用域錯誤，應移入 customRef 工廠

注意這個代理人**在同一份報告下半段又寫**：

> 不過以目前的程式碼，因為 `timer` 在外層，但每個 `useDebouncedRef` 呼叫會建立各自的 closure，所以「不同 `useDebouncedRef` 呼叫之間」其實還是隔離的。

**自己寫 Critical、自己又承認其實不會 bug、嚴重度卻沒下修**。這是純粹「找問題湊字數」的副作用 %( ´_ゝ`)%

而詳細 baseline 對同一個 case 展現自我修正能力，原本標 Critical 後又重新評估：

> 重新評估後此項實際不會觸發 bug，降級為 Low。

這就是**要求 prompt 內寫「Critical 必須能描述觸發路徑」的威力**，強制代理人對 Critical 自我審查。

但即使詳細 baseline 修掉 Critical，3 個 High（其實該是 Medium）仍標出來，膨脹仍在。

### 額外發現品質：三者旗鼓相當

那「額外發現」的價值呢？

三組都抓到了那個經典，**Case B 估時加總 4 天卻寫「3 天」**的算術錯誤。三組也都點出 SSR 相容、source 型別契約、命名違規等延伸。**這部分沒有顯著差異**。

額外發現品質似乎主要取決於「代理人 base model」而不是「prompt 設計」。Opus 4.7 對 Vue 生態足夠熟，三種 prompt 都能挖到類似深度的延伸。

### 結論：skill 的真正價值是什麼？

把四個指標放在一起看：

| 指標 | Skill | 詳細 | 籠統 |
| ---- | ----- | ---- | ---- |
| 預埋命中率 | 100% | 100% | 56–80% |
| Case D Critical 誤報 | 0 | 0（自我修正） | **1 個** |
| 額外發現品質 | 高 | 高 | 高 |
| 嚴重度膨脹 | 略過度 | 仍膨脹 | 嚴重膨脹 + 自相矛盾 |

真相是：

1. **找更多預埋問題不是 skill 的核心優勢**：寫得夠用力的 prompt 一樣 100% 命中
2. **降低 Critical 誤報才是真正的差距**：籠統 baseline 直接誤報 Critical，隨手丟一句「請審查」就是這個下場
3. **正交視角封裝才是 skill 的本質**：把「列檢查清單 + 分視角 + 禁讀答案本 + Critical 自我審查」這套技巧**封裝成可重複呼叫的流程**，使用者不用每次自己寫

換句話說，**`independent-review` skill 不是讓你找到更多 bug，而是讓你每次都用同一套高品質的 prompt，不會偷懶**。

```text
路人：「所以 skill 是高品質 prompt 的捷徑？」
鱈魚：「對，而且它強制你不能寫籠統 prompt。一寫籠統，Critical 誤報就會打你臉 %( ´_ゝ`)%」
```

這也解釋了為什麼平常隨手「請 Claude 幫我看一下這段程式碼」常會踩雷，**人在偷懶時就會用籠統風格，skill 把懶人路徑堵起來了** 🐟

## 總結 🐟

跑完這 4 案 8 位代理人，我的結論是：

- **`independent-review` skill 確實有效**：預埋問題 100% 命中，視角正交設計奏效，整合產出可用
- **代理人偏向**：嚴重度會膨脹一級、測試覆蓋率視角永遠覺得不夠、乾淨案例會被湊問題
- **對照組是真正的試金石**：沒它你看不到誤報率，會以為代理人永遠正確
- **整合階段不能省**：照單全收很危險，要做一次嚴重度校準與去重
- **額外發現常比預埋問題更值得**：盲點掃除是 skill 真正的價值
- **測試自己的 skill 本身就是 skill 的副產品**：跑這次測試讓我更清楚 prompt 哪裡可以加強
