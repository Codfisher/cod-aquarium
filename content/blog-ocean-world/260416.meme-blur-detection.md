# 梗圖太糊怎麼辦？自動模糊偵測的踩坑之旅 %(・∀・)９%

問題是圖庫裡有近 900 張圖，總不能一張一張人工檢查吧？%(◞‸◟ )%

鱈魚：「我需要一個演算法來幫我判斷圖片的模糊程度！」

路人：「聽起來很簡單啊，不就是算個分數嗎？」

鱈魚：「我一開始也這麼想......結果踩了一堆坑 %(´;ω;`)%」

## Laplacian Variance：最經典的方案

在圖像處理領域，判斷模糊度最常見的方法是 **Laplacian Variance**。

原理其實不複雜：

1. 將圖片轉為灰階
2. 對每個像素套用 Laplacian 核心（一種邊緣偵測的卷積運算）
3. 計算結果的變異數

變異數越高代表邊緣越多、圖片越清晰；越低則越模糊。

Laplacian 核心長這樣：

```
 0  1  0
 1 -4  1
 0  1  0
```

簡單說就是拿中心像素乘 -4，加上上下左右四個鄰居，如果周圍跟中心差不多（平滑區域），結果趨近 0；如果有明顯邊緣，結果會很大。

用 Node.js 搭配 sharp 實作：

```ts
async function computeBlurScore(filePath: string) {
  const { data, info } = await sharp(filePath)
    .grayscale()
    .raw()
    .toBuffer({ resolveWithObject: true })

  const { width, height } = info
  let sum = 0
  let sumSquare = 0
  let count = 0

  for (let y = 1; y < height - 1; y++) {
    for (let x = 1; x < width - 1; x++) {
      const index = y * width + x
      const laplacian
        = -4 * data[index]
          + data[(y - 1) * width + x]
          + data[(y + 1) * width + x]
          + data[y * width + (x - 1)]
          + data[y * width + (x + 1)]

      sum += laplacian
      sumSquare += laplacian * laplacian
      count++
    }
  }

  const mean = sum / count
  const variance = sumSquare / count - mean * mean
  return Math.round(variance * 100) / 100
}
```

對 885 張梗圖跑了一次，結果看起來不錯：

| 指標 | 數值 |
|------|------|
| 平均分數 | 936 |
| 中位數 | 494 |
| 最低（最糊）| 3.69 |
| 最高（最清晰）| 18,671 |

分數低於 50 的有 48 張，看起來確實是很糊的圖片。

鱈魚：「完美！任務完成！ %(ゝ∀・)b%」

路人：「真的嗎？」

## 文字疊加的陷阱

把分數接上前端 UI 後，實際使用才發現不對勁。

有些圖片**明明很糊**，分數卻高得離譜。

仔細一看，這些圖有個共同特徵：**上面有文字**。

梗圖最常見的組成就是「一張圖 + 一段文字」，而文字的邊緣極為銳利（尤其白字黑邊），Laplacian 會產生極高的響應值，直接把整體變異數拉上去。

跑了一下數據驗證：

```
有文字的圖平均分數：1001.6（616 張）
無文字的圖平均分數：653.7（269 張）
```

前 15 高分的圖片中，有 11 張含文字。

鱈魚：「所以一張超糊的圖，只要上面貼幾行字，分數就飆上去了？」

路人：「沒錯，變異數就是這麼容易被極端值帶風向 %(╯°Д°)╯︵ ┻━┻%」

## 嘗試中位數：矯枉過正

既然變異數會被極端值影響，那改用**中位數**呢？

中位數取的是「排序後正中間的值」，少數文字像素的極端 Laplacian 值不會影響結果。

聽起來很合理，改完跑了一次......

```
分數分布：
  score=0: 187 張 (21%)
  score=1: 317 張 (35%)
  score=2: 213 張 (24%)
  score=3:  70 張 (7%)
```

80% 的圖片分數集中在 0~2。

鱈魚：「......這跟沒算一樣吧 %(´;ω;`)%」

問題出在 Laplacian 絕對值本身就是整數，大部分像素在平滑區域的值為 0 或 1，中位數直接落在這些低值上，完全喪失區分度。

## P75：找到平衡點

中位數太穩定，變異數太極端，那取個折衷。

**第 75 百分位數（P75）** 跳過大面積的平滑區域（零值），同時不會被少數極端值綁架。

用幾張已知模糊和清晰的圖片做對比：

| 圖片類型 | Variance | Median | P75 |
|---|---|---|---|
| 模糊 | 3.7 | 1 | 2 |
| 中等品質 | 115.9 | 2 | 6 |
| 文字+模糊底圖 | 9,656 | 1 | 37 |
| 清晰 | 9,968 | 31 | 95 |

P75 成功讓文字糊圖（37）和真正清晰的圖（95）拉開差距，而 Variance 根本分不開（9,656 vs 9,968）。

不過問題又來了。

## 漫畫與線稿：P75 的盲點

使用者回報了一張漫畫截圖，看起來明顯模糊，但 P75 分數有 51。

原因是黑白漫畫的**線稿筆觸本身就是銳利的邊緣**，跟文字一樣會產生高 Laplacian 響應。只要圖片中有超過 25% 的像素落在線條上，P75 就會被撐上去。

鱈魚：「怎麼每次以為解決了，就冒出新的坑 %(;´༎ຶД༎ຶ`)%」

## 縮圖 + P75：雙管齊下

既然文字和線條會干擾，那**先縮小圖片**呢？

縮圖時，細小的文字邊緣和線條會在重採樣過程中被模糊掉，但圖片整體的清晰度特徵會保留。

測試了不同的縮圖尺寸，200px 是最佳平衡：

| 圖片類型 | 純 P75 | 縮圖 + P75 |
|---|---|---|
| 模糊 | 2 | 3 |
| 西裝糊圖 | 15 | 9 |
| 漫畫 | 51 | **37** |
| 文字+模糊 | 37 | **55** |
| 清晰 | 72~95 | **58~105** |

漫畫從 51 降到 37，改善不少。但隨後又發現一張「銳利線稿合成模糊人臉」的圖片，分數仍有 44。

## 最終方案：縮圖 + 分塊 + P25

最後的殺手鐧是**分塊計算**。

核心想法是：把圖片切成 4×4 共 16 個區塊，各自計算 P75，然後取這 16 個分數的**第 25 百分位數**。

為什麼取 P25？因為我們想反映的是圖片中**較弱區域**的品質。如果某些區塊有銳利的文字或線條，它們的分數會很高，但不會影響 P25 抓到的「較弱區塊」。

結合縮圖一起用：

1. **縮圖至 200px**：降低文字與線稿的干擾
2. **切成 4×4 區塊**：各自計算 Laplacian P75
3. **取區塊分數的 P25**：反映較弱區域的品質

```ts
const BLUR_SCORE_RESIZE = 200
const BLUR_SCORE_GRID = 4

async function computeBlurScore(filePath: string) {
  const { data, info } = await sharp(filePath)
    .resize(BLUR_SCORE_RESIZE, BLUR_SCORE_RESIZE, {
      fit: 'inside',
      withoutEnlargement: true,
    })
    .grayscale()
    .raw()
    .toBuffer({ resolveWithObject: true })

  const { width, height } = info
  const grid = BLUR_SCORE_GRID
  const blockWidth = Math.floor(width / grid)
  const blockHeight = Math.floor(height / grid)
  const blockScoreList: number[] = []

  for (let gy = 0; gy < grid; gy++) {
    for (let gx = 0; gx < grid; gx++) {
      const score = computeBlockP75(
        data,
        width,
        gx * blockWidth,
        gy * blockHeight,
        blockWidth,
        blockHeight,
      )
      blockScoreList.push(score)
    }
  }

  blockScoreList.sort((a, b) => a - b)
  const p25Index = Math.floor(blockScoreList.length / 4)
  return blockScoreList[p25Index] ?? 0
}
```

其中 `computeBlockP75` 使用**計數排序**找 P75，因為 Laplacian 絕對值的範圍固定為 0~1020，直接用一個 1021 長度的陣列當計數桶，不需要通用排序：

```ts
function computeBlockP75(
  data: Buffer,
  width: number,
  startX: number,
  startY: number,
  blockWidth: number,
  blockHeight: number,
) {
  // Laplacian 絕對值最大為 4×255 = 1020
  const bucketList = new Uint32Array(1021)
  let total = 0

  for (let y = startY + 1; y < startY + blockHeight - 1; y++) {
    for (let x = startX + 1; x < startX + blockWidth - 1; x++) {
      const index = y * width + x
      const value = Math.abs(
        -4 * data[index]
        + data[(y - 1) * width + x]
        + data[(y + 1) * width + x]
        + data[y * width + (x - 1)]
        + data[y * width + (x + 1)],
      )
      bucketList[value]++
      total++
    }
  }

  const p75Target = Math.floor(total * 0.75)
  let cumulative = 0
  for (let i = 0; i < bucketList.length; i++) {
    cumulative += bucketList[i]
    if (cumulative > p75Target)
      return i
  }
  return 0
}
```

最終結果：

| 圖片類型 | Variance | 純 P75 | 縮圖+P75 | **縮圖+分塊 P25** |
|---|---|---|---|---|
| 模糊 | 3.7 | 2 | 3 | **2** |
| 西裝糊圖 | 399 | 15 | 9 | **14** |
| 漫畫糊圖 | 3,431 | 51 | 37 | **19** |
| 文字+糊底 | 9,657 | 37 | 55 | **24** |
| 清晰 | 9,968 | 95 | 58 | **44** |
| 清晰 | 7,233 | 72 | 105 | **108** |

漫畫從最初的 3,431（Variance）降到 19，文字糊圖從 9,657 降到 24，與清晰圖片（44~108）有了明確的差距。

## 接上前端 UI

有了分數，就能在前端做過濾了。在選單加入一個 4 段 slider，對應不同閾值：

| 等級 | 閾值 | 剩餘圖片 | 比例 |
|---|---|---|---|
| 全部 | 0 | 885 | 100% |
| 略糊 | ≥ 5 | 599 | 67% |
| 普通 | ≥ 15 | 262 | 29% |
| 清晰 | ≥ 30 | 104 | 11% |

實作上把搜尋結果和過濾拆開：`searchedList` 存搜尋結果，`filteredList` 是套用模糊過濾後的 computed，這樣調整 slider 時不會重新觸發搜尋或打亂排序。

## 已知限制

這個方案仍有天然限制：

- **銳利線稿合成模糊內容**：如果線條均勻分布在每個區塊，分塊也無法排除。這類圖需要語意理解（例如人臉偵測）才能正確判斷
- **純色/漸層圖片**：沒有邊緣就沒有分數，會被歸類為模糊。不過梗圖很少出現這種情況

## 總結 🐟

- Laplacian Variance 是經典方案，但容易被文字疊加和線稿干擾
- 中位數抗干擾但區分度不足，P75 是比較好的折衷
- **縮圖**能有效模糊掉文字邊緣，降低誤判
- **分塊計算**讓局部模糊被正確反映，不被銳利區域拉高分數
- 最終方案「縮圖 200px + 4×4 分塊 P75 + 取 P25」在大多數情況下表現良好
- 使用**計數排序**找百分位數，避免大陣列排序的效能問題

不過最終的方案在 885 張梗圖上跑得很穩，演算法本身也很輕量，不需要引入 OpenCV 或機器學習模型，只靠 sharp 就搞定了 %(・∀・)９%
