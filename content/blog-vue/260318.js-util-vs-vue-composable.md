# JS util vs Vue Composable，阿不就都是 JS Function 嘛？(´・ω・`)

熟悉 Vue 3 的朋友們，應該都寫過 Composition API 吧？

之前有聽過有人問：「為什麼我要寫個 `useXxx` 的 composable，而不是直接寫個普通的 util function？」

其實 JS util 和 Vue Composable 本質上的確都是 function，但 Composable 有一些重要的約定。

今天就來聊聊這兩者的差異，這也是目前 AI 最常犯的錯誤。

## Utility Function 就是純 JS function

先來看最單純的情況。Utility function 就是一般的 JavaScript function，負責做資料轉換、格式化、驗證等等純粹的運算。

例如：

```ts
export function formatDate(date: Date) {
  return new Intl.DateTimeFormat('zh-TW').format(date)
}

export function formatCurrency(amount: number) {
  return new Intl.NumberFormat('zh-TW', {
    style: 'currency',
    currency: 'TWD',
  }).format(amount)
}

export function validateEmail(email: string) {
  return /^[^\s@]+@[^\s@][^\s.@]*\.[^\s@]+$/.test(email)
}
```

這類 function 有幾個特點：

- **沒有狀態**：輸入相同，輸出一定相同
- **不依賴 Vue**：拿去 Node.js、React 裡面用也完全沒問題
- **隨時隨地都能呼叫**：在 `setup`、`onMounted`、`watch`、甚至 `setTimeout` 裡面呼叫都可以

重點是，這些 function 不需要加 `use` 前綴，也不需要包成 composable。

## Composable 不只是 function

Composable 雖然也是 function，但它會使用 Vue 的響應式系統和生命週期。

```ts
import { onMounted, onUnmounted, ref } from 'vue'

export function useWindowSize() {
  const width = ref(window.innerWidth)
  const height = ref(window.innerHeight)

  function update() {
    width.value = window.innerWidth
    height.value = window.innerHeight
  }

  onMounted(() => window.addEventListener('resize', update))
  onUnmounted(() => window.removeEventListener('resize', update))

  return { width, height }
}
```

這時候你可能會想問：「Vue 的響應式系統本來就可以獨立運作，就算不是 Vue 也可以用，所以分界線在哪？」

主要差別是 Composable 內部會使用 `onMounted`、`onUnmounted` 等 lifecycle hook，這些 hook 必須在**元件實例的同步流程**中才能正確註冊。

## Composable 必須在同步流程中呼叫

Vue 在執行 `<script setup>` 時，會將目前的元件實例設為 current instance。只要你還在這個同步的呼叫堆疊中，lifecycle hook 就能正確註冊到元件上。

所以以下這些地方呼叫 composable 都是可以的：

```vue
<script setup lang="ts">
import { onMounted, ref, watch } from 'vue'
import { useWindowSize } from './composables/use-window-size'

// ✓ script setup 頂層
const { width, height } = useWindowSize()

// ✓ onMounted callback（仍在同步流程中）
onMounted(() => {
  const size = useWindowSize()
})

// ✓ watch callback（仍在同步流程中）
const enabled = ref(false)
watch(enabled, () => {
  const size = useWindowSize()
})
</script>
```

因為 `onMounted`、`watch` 的 callback 執行時，Vue 仍然會將元件實例設為 current instance，所以裡面的 lifecycle hook 都能正確註冊。%╰(*´︶`*)╯%

但是，一旦進入**非同步流程**，元件實例就不再是 current instance 了：

```vue
<script setup lang="ts">
import { onMounted } from 'vue'
import { useWindowSize } from './composables/use-window-size'

onMounted(() => {
  // ✗ setTimeout 是非同步，脫離了元件的同步流程
  setTimeout(() => {
    const size = useWindowSize()
  }, 100)

  // ✗ Promise.then 也是非同步
  fetch('/api').then(() => {
    const size = useWindowSize()
  })
})

// ✗ await 之後也是非同步
const data = await fetchSomething()
const size = useWindowSize()
</script>
```

`setTimeout`、`Promise.then`、`await` 之後的程式碼，都已經脫離了元件實例的同步呼叫堆疊，lifecycle hook 無法正確註冊，自動清除機制就會壞掉。

意思就是在非同步流程中呼叫 lifecycle hook 會導致 memory leak。

## 那 util 呢？隨便你放

相比之下，util function 沒有這些限制，因為它們根本不碰 Vue 的響應式系統和生命週期。

```vue
<script setup lang="ts">
import { onMounted, ref, watch } from 'vue'
import { formatDate } from './utils/formatters'
import { validateEmail } from './utils/validators'

const email = ref('')
const today = ref('')

// ✓ 在頂層呼叫
today.value = formatDate(new Date())

// ✓ 在 onMounted 呼叫
onMounted(() => {
  console.log(formatDate(new Date()))
})

// ✓ 在 watch 呼叫
watch(email, (value) => {
  if (validateEmail(value)) {
    console.log('Email 合法')
  }
})

// ✓ 甚至在 setTimeout 呼叫
setTimeout(() => {
  console.log(formatDate(new Date()))
}, 1000)
</script>
```

全部都可以，因為它們就是普通的 function，沒有副作用，也不依賴 Vue 的元件實例。

## 不要把 util 包成 composable

有時候會看到有人把明明是 util 的東西，硬是包成 composable 的格式：

```ts
// ✗ 不需要這樣做
export function useFormatters() {
  const formatDate = (date: Date) => {
    return new Intl.DateTimeFormat('zh-TW').format(date)
  }

  const formatCurrency = (amount: number) => {
    return new Intl.NumberFormat('zh-TW', {
      style: 'currency',
      currency: 'TWD',
    }).format(amount)
  }

  return { formatDate, formatCurrency }
}
```

這樣做完全沒有好處，反而帶來壞處：

- 多了一層不必要的包裝
- `use` 前綴讓人誤以為裡面有響應式狀態
- 使用時多了解構的步驟

直接 export function 就好了，簡單明瞭。%( ˘ω˘ )%

## Composable 搭配 Util 一起用

不過 composable 和 util 不是互斥的。composable 裡面當然可以使用 util function：

```ts
// utils/validators.ts
// composables/use-email-input.ts
import { computed, ref } from 'vue'
import { validateEmail } from '../utils/validators'

export function validateEmail(email: string) {
  return /^[^\s@]+@[^\s@][^\s.@]*\.[^\s@]+$/.test(email)
}

export function useEmailInput(initialValue = '') {
  const email = ref(initialValue)
  const isValid = computed(() => validateEmail(email.value))
  const error = computed(() =>
    email.value && !isValid.value ? '無效的 Email 格式' : null
  )

  return { email, isValid, error }
}
```

`validateEmail` 是純粹的 util，`useEmailInput` 才是 composable。各司其職，結構清晰。%( •̀ ω •́ )✧%

## 用 useIntervalFn 來感受差異

光看文字可能還是有點抽象，讓我們用 VueUse 的 `useIntervalFn` 來實際感受一下「同步 vs 非同步呼叫」的差別。

先來看看 `useIntervalFn` 的簡化版原始碼，理解它做了什麼：

```ts
import { onUnmounted, ref } from 'vue'

export function useIntervalFnSimple(
  callback: () => void,
  interval: number,
) {
  let timerId: ReturnType<typeof setInterval> | null = null
  const isActive = ref(false)

  function stop() {
    if (timerId) {
      clearInterval(timerId)
      timerId = null
    }
    isActive.value = false
  }

  function start() {
    stop()
    timerId = setInterval(callback, interval)
    isActive.value = true
  }

  start()
  onUnmounted(stop)

  return { isActive, start, stop }
}
```

重點在最後那行 `onUnmounted(stop)`。它會在元件銷毀時自動呼叫 `stop()`，interval 就會跟著被清除。但前提是 `onUnmounted` 要在正確的時機被呼叫，也就是在元件實例的同步流程中。

接下來讓我們用實際的元件來對比看看。兩邊都是每秒 +1 的計數器，差別只在 `useIntervalFn` 的呼叫位置。

點「銷毀元件」後觀察計數器的變化：

<interval-demo />

左邊在同步流程中呼叫，銷毀後計數停止。右邊在 `setTimeout`（非同步）中呼叫，銷毀後計數還在跑。同樣的 composable，只是呼叫時機不同，結果天差地別。%( ˘•ω•˘ )%

### 同步流程中呼叫：自動清除 %(´▽`)%

```vue
<script setup lang="ts">
import { ref } from 'vue'
import { useIntervalFnSimple } from './use-interval-fn-simple'

const count = ref(0)

// ✓ 在同步流程中呼叫
// onUnmounted 正確綁定到元件實例
// 元件銷毀時 interval 自動停止
useIntervalFnSimple(() => {
  count.value++
}, 1000)
</script>
```

在 `onMounted` callback 裡呼叫也一樣沒問題，因為仍然在元件的同步流程中：

```vue
<script setup lang="ts">
import { onMounted, ref } from 'vue'
import { useIntervalFnSimple } from './use-interval-fn-simple'

const count = ref(0)

// ✓ onMounted callback 仍在同步流程中
onMounted(() => {
  useIntervalFnSimple(() => {
    count.value++
  }, 1000)
})
</script>
```

元件銷毀時，`onUnmounted` 會自動觸發 `stop()`，interval 乾淨地被清除。你完全不用操心任何清除邏輯。%╰(*´︶`*)╯%

### 非同步流程中呼叫：自動清除失效 %(╥ω╥`)%

```vue
<script setup lang="ts">
import { onMounted, ref } from 'vue'
import { useIntervalFnSimple } from './use-interval-fn-simple'

const count = ref(0)

// ✗ 在非同步 callback 裡呼叫
// setTimeout、Promise.then、await 之後都算非同步
setTimeout(() => {
  // 此時已經脫離元件實例的同步流程
  // onUnmounted 無法正確註冊，元件銷毀後 interval 不會被清除！
  useIntervalFnSimple(() => {
    count.value++
  }, 1000)
}, 100)
</script>
```

`setTimeout` callback 執行時，已經脫離了元件實例的同步呼叫堆疊。這時候 `useIntervalFnSimple` 內部的 `onUnmounted` 無法正確註冊到元件上，元件銷毀後 interval 不會被自動清除，就像上面的 demo 一樣，數字會持續增加。

同樣的道理也適用在 `Promise.then`、`await` 之後呼叫 composable，只要脫離了同步流程，`onUnmounted` 都無法正確註冊。

這就是為什麼 composable 必須在元件的同步流程中呼叫，否則 lifecycle hook 無法正確註冊，導致 memory leak。

不過其實 Vue 在 dev 模式下會貼心地給你一個警告：

```bash
[Vue warn]: onUnmounted is called when there is no active component
instance to be associated with.
Lifecycle injection APIs can only be used during execution of setup().
If you are using async setup(), make sure to register lifecycle hooks
before the first await statement.
```

所以如果你在 console 看到這段訊息，就表示有 composable 在非同步流程中被呼叫了，趕快檢查一下吧。%( •̀ ω •́ )✧%

## 總結 🐟

- JS Utility Function 是純粹的 JavaScript function，不依賴 Vue，隨時隨地都能呼叫
- Vue Composable 會使用響應式系統（`ref`、`computed`）和生命週期（`onMounted` 等），必須在**元件實例的同步流程**中呼叫
- `<script setup>` 頂層、`onMounted`、`watch` callback 都屬於同步流程，可以安全呼叫 composable
- `setTimeout`、`Promise.then`、`await` 之後屬於非同步流程，lifecycle hook 無法正確註冊，會導致 memory leak
- 不要把純粹的 util function 包成 composable，`use` 前綴應該保留給真正有響應式狀態的 function
- Composable 和 Util 可以搭配使用，各司其職
