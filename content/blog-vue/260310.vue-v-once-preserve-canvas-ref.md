# 你的 DOM 不是原來的 DOM

大家好，我是鱈魚。(｡･∀･)ﾉﾞ

最近在開發 [CodStack](/aquarium/codstack/) 時，遇到一個詭異的問題：明明 `canvas` 有綁 `ref`，Babylon.js 也正常初始化，但只要開啟右鍵選單，原本註冊在 canvas 的事件就失效了。

讓我們來研究研究。

## 重現問題

我把 3D 場景初始化的內容放在 `use-babylon-simple` composable 中，簡化後的結構如下：

```typescript
export function useBabylonSimple(
  canvasRef: MaybeRefOrGetter<HTMLCanvasElement | null>,
) {
  let engine: Engine | null = null;
  let scene: Scene | null = null;

  // 省略初始化過程

  const canvasSize = reactive(useElementSize(canvasRef));
  watch(canvasSize, () => {
    engine?.resize();
  });
}
```

其中 `useElementSize` 偵測 canvas 的尺寸變化，並在尺寸變化時調用 `engine.resize()` 方法。

<auto-resize />

可以注意到不管你怎麼調整寬度，場景會自動調整尺寸，方塊不會扁掉。%◝( •ω• )◟%

接下來讓我們把 `canvas` 放進 Nuxt UI 的 `UContextMenu` 元件中，讓我們可以開啟右鍵選單。

```vue
<template>
  <u-context-menu :items="contextMenuItems">
    <canvas ref="canvasRef" class="w-full h-full outline-none block" />
  </u-context-menu>
</template>
```

<content-menu />

現在會發現鬼故事發生了，**開啟右鍵選單前**改變寬度一切正常，但**開啟右鍵選單後**改變寬度會發現方塊扁掉了！%╭(°A ,°`)╮%

## 為什麼會壞掉？

雖然不確定 `UContextMenu` 內部做了什麼，但關鍵可能在於 Vue 的更新機制。

當元件重新渲染時，Vue 會重新執行 render function，產生新的 VNode 樹，接著透過 patch 比對新舊 VNode 來更新 DOM。

因為 **slot 內容屬於父元件的 render scope**，所以當父元件重新渲染時，slot 內容也會被重新渲染。

也就是說，選單開啟、父元件重新渲染時，slot 裡面的 `<canvas>` 也會被重新評估。

一般情況下 Vue 的 diff 演算法不會真的銷毀再重建相同的 DOM 節點，但 `ContextMenuTrigger` 在狀態變化時重新渲染，可能導致 slot 內的 canvas DOM 被替換。

一旦 canvas DOM 被替換，`ref` 會自動指向新建立的 canvas 元素。而 Babylon.js 的渲染迴圈（render loop）持有的是初始化時取得的 canvas 參考，所以場景本身依舊正常顯示。

奇怪的是 `useElementSize`，其內部使用 `useResizeObserver`，從原始碼來看，`useResizeObserver` 有處理 DOM Ref 更新的情境，不知道為甚麼 `useResizeObserver` 偵測就是會失效，導致：

1. **`useElementSize` 不再響應尺寸變化**：使 `engine.resize()` 不再被觸發
2. **畫面比例失真**：引擎持續以舊的畫布尺寸渲染，但 CSS 已經改變了 canvas 的顯示大小，方塊就被拉伸變形了

如果各位大大知道底層到底發生甚麼事，還請不吝分享。%( ´∀`)~♥%

## 如何除靈

所以要怎麼除靈呢？只能改變 template 結構嗎？的確也可以，只是有一個更簡單的解決方案。%( •̀ ω •́ )✧%

那就是出場次數堪比日本製壓縮機，聊天紀錄永遠輪迴在「下次約」，我們那個極度少見的朋友：**`v-once`**！%ヽ(•̀∀•̀ヽ )%

<br>

路人：「你這樣是歧視 v-once。%ლ(╹ε╹ლ)%」

鱈魚：「不然你問各位讀者，有用過的舉手，我數數看 %(´・ω・`)%」

<br>

`v-once` 的作用是告訴 Vue：「這個節點只渲染一次，之後的更新都跳過」。

加上 `v-once` 後，Vue 在首次渲染時會正常建立 canvas DOM 並賦值給 `canvasRef`，但在後續的 patch 過程中，這個 VNode 會被標記為靜態節點，直接跳過比對。

這樣不管 `UContextMenu` 怎麼更新，canvas 的 DOM 永遠是同一個，`ref` 指向也不會改變。

用法很簡單，就是加上一個 `v-once` 指令。

```vue
<template>
  <context-menu :items="contextMenuItems">
    <canvas v-once ref="canvasRef" class="w-full h-full outline-none block" />
  </context-menu>
</template>
```

以下是使用 `v-once` 前後的對比，可以看到使用 `v-once` 後，即可開啟選單且不管怎麼調整寬度，方塊都不會扁掉。%(\*´∀`)~♥%

## 什麼時候適合用 v-once

`v-once` 並不是萬用解法，但在以下情境非常實用：

- **第三方函式庫接管的 DOM 元素**：canvas（WebGL）、地圖容器、影片播放器等
- **不需要 Vue 響應式更新的靜態內容**：一次性渲染的大量列表項目
- **放在會頻繁更新的父元件 slot 裡的穩定元素**：就像本文的案例

反過來說，如果元素本身的 attribute 或內容需要隨資料變化，就不適合用 `v-once`，因為加上去之後這些綁定就不會再更新了。

如同那些被作者棄坑的作品，再也不會更新了。%(╥ω╥`)%

## 總結 🐟

- 當 `canvas` 放在會頻繁更新的元件 slot 內，Vue 的重新渲染可能導致 canvas DOM 被替換，使 `ref` 指向失效
- `v-once` 可以讓 Vue 跳過該節點的後續更新，確保 DOM 元素不被替換
- 特別適合用在由第三方函式庫（Babylon.js、Three.js 等）接管的 DOM 元素上
- 注意 `v-once` 會讓該元素上所有的動態綁定失效，只適合用在不需要 Vue 更新的元素
