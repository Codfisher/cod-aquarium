# 用 v-once 保護 canvas 不被 Vue 重新渲染搞壞

大家好，我是鱈魚。(｡･∀･)ﾉﾞ

最近在開發 3D 場景編輯器時，遇到一個詭異的問題：明明 `canvas` 有綁 `ref`，Babylon.js 也正常初始化了，但只要右鍵選單的選項一更新，整個 canvas 就壞掉了。

各種 DOM 相關的 API 全部失效，滑鼠事件不響應、渲染也停了。(´;ω;`)

排查後發現，只要在 `<canvas>` 上加一個 `v-once` 就解決了。讓我們來聊聊背後的原因。

## 問題場景

先來看看簡化後的結構：

```vue
<template>
  <context-menu :items="contextMenuItems">
    <canvas ref="canvasRef" class="w-full h-full outline-none block" />
  </context-menu>
</template>
```

`<canvas>` 被放在 `<context-menu>` 元件的 slot 裡面，而 `contextMenuItems` 是一個會根據「目前選取的物件」動態變化的響應式資料。

看起來很正常對吧？但每次 `contextMenuItems` 一更新，canvas 就會出事。

## 為什麼會壞掉？

關鍵在於 Vue 的更新機制。

當父元件的響應式資料變化時，Vue 會重新執行 render function，產生新的 VNode 樹，接著透過 patch 比對新舊 VNode 來更新 DOM。

問題在於：**slot 內容屬於父元件的 render scope**。

也就是說，當 `contextMenuItems` 改變，導致父元件重新渲染時，slot 裡面的 `<canvas>` 也會被重新評估。

一般情況下 Vue 的 diff 演算法不會真的銷毀再重建相同的 DOM 節點。但在實際場景中，像 Nuxt UI 的 `UContextMenu` 這類元件，內部會根據 `items` 的結構變化產生不同的條件分支、動態 slot 數量等，導致包裹 default slot 的子樹 key 或結構改變，Vue 的 patch 會認定這是不同的子樹，進而**銷毀並重建整個 slot 內容**。

一旦 canvas DOM 被替換：

1. **`ref` 指向新的空 canvas**：原本 Babylon.js 綁定的那個 canvas 已經不在 DOM 上了
2. **WebGL context 遺失**：渲染引擎綁定的是舊的 canvas，新的 canvas 沒有 WebGL context
3. **所有事件監聽失效**：pointer 事件、resize 等都註冊在舊 canvas 上

簡單來說就是：引擎還在操作舊的 canvas，但畫面上顯示的已經是一個全新的、空白的 canvas 了。

## v-once 如何解決

```vue
<template>
  <context-menu :items="contextMenuItems">
    <canvas v-once ref="canvasRef" class="w-full h-full outline-none block" />
  </context-menu>
</template>
```

`v-once` 的作用是告訴 Vue：「這個節點只渲染一次，之後的更新都跳過」。

加上 `v-once` 後，Vue 在首次渲染時會正常建立 canvas DOM 並賦值給 `canvasRef`，但在後續的 patch 過程中，這個 VNode 會被標記為靜態節點，直接跳過比對。

這樣不管 `contextMenuItems` 怎麼更新，canvas 的 DOM 永遠是同一個，`ref` 指向也不會改變。

::: tip
可以把 `v-once` 想像成對 Vue 說：「這個元素我自己管，你不要碰。」非常適合用在需要由第三方函式庫（如 Babylon.js、Three.js、Chart.js 等）接管的 DOM 元素上。
:::

## 實際案例

以下是我的 3D 場景編輯器中的實際程式碼：

```vue
<context-menu
  :preview-mesh="previewMesh"
  :selected-meshes="selectedMeshes"
  :added-mesh-list="addedMeshList"
  :can-redo="canRedo"
  :can-undo="canUndo"
  v-on="contentMenuEvents"
>
  <canvas
    v-once
    ref="canvasRef"
    class="w-full h-full outline-none block"
  />
</context-menu>
```

`<context-menu>` 接收了大量的響應式 props（預覽模型、選取清單、undo/redo 狀態等），這些資料隨時都在變化。如果沒有 `v-once`，canvas 就會在這些更新中被波及。

## 什麼時候適合用 v-once

`v-once` 並不是萬用解法，但在以下情境非常實用：

- **第三方函式庫接管的 DOM 元素**：canvas（WebGL）、地圖容器、影片播放器等
- **不需要 Vue 響應式更新的靜態內容**：一次性渲染的大量列表項目
- **放在會頻繁更新的父元件 slot 裡的穩定元素**：就像本文的案例

反過來說，如果元素本身的 attribute 或內容需要隨資料變化，就不適合用 `v-once`，因為加上去之後這些綁定就不會再更新了。

## 總結 🐟

- 當 `canvas` 放在會頻繁更新的元件 slot 內，Vue 的重新渲染可能導致 canvas DOM 被替換，使 `ref` 指向失效
- `v-once` 可以讓 Vue 跳過該節點的後續更新，確保 DOM 元素不被替換
- 特別適合用在由第三方函式庫（Babylon.js、Three.js 等）接管的 DOM 元素上
- 注意 `v-once` 會讓該元素上所有的動態綁定失效，只適合用在不需要 Vue 更新的元素
