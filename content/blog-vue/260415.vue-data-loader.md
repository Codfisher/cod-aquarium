## 什麼是 fetch 瀑布流？

先來看一個實務上極為常見的文章詳細頁場景。頁面有三層巢狀元件，所需的 ID 明明都在網址上，卻因為 UI 的畫面載入邏輯相互牽累：

```vue
<!-- PostPage.vue -->
<script setup lang="ts">
import { useAsyncState } from '@vueuse/core'
import { useRoute } from 'vue-router'

const route = useRoute()
const postId = Number(route.params.id) // 從網址取得 postId

const { state: post } = useAsyncState(() => fetchPost(postId), undefined) // 需耗時 1 秒
</script>

<template>
  <!-- 實務上往往為了顯示 Loading 畫面而使用外層 v-if -->
  <div v-if="post">
    <h1>{{ post.title }}</h1>

    <!-- ❌ 留言區明明只需 postId 就能抓資料，卻被外層 v-if 拖累，白白等上層 1 秒才開始掛載！ -->
    <post-comment-list :post-id="postId" />
  </div>
  <div v-else>
    載入中⋯
  </div>
</template>
```

```vue
<!-- PostCommentList.vue -->
<script setup lang="ts">
import { useAsyncState } from '@vueuse/core'

const props = defineProps<{ postId: number }>()

const { state: commentList } = useAsyncState(
  () => fetchCommentList(props.postId), // 再等 1 秒
  undefined,
)
</script>

<template>
  <!-- 留言載入時，一樣用 v-if 處理狀態 -->
  <ul v-if="commentList">
    <li v-for="comment in commentList" :key="comment.id">
      {{ comment.content }}
    </li>

    <!-- ❌ 相似推薦文章也只需要 postId，卻又被上一層的 v-if 取代順位給卡死！ -->
    <post-related-list :post-id="postId" />
  </ul>
</template>
```

```vue
<!-- PostRelatedList.vue -->
<script setup lang="ts">
import { useAsyncState } from '@vueuse/core'

const props = defineProps<{ postId: number }>()

const { state: relatedList } = useAsyncState(
  () => fetchRelatedPosts(props.postId), // 又等 1 秒
  [],
)
</script>
```

你看，這正是最常見的雷區。很多時候 API 明明可以同時發起，但只要子元件被 `v-if`（或 `Suspense`）擋住，**當父元件還沒拿到資料時，子元件根本無從建立**，瀑布流就這樣產生了：

```
User     ████████░░░░░░░░░░░░░░░░  1s
Post     ░░░░░░░░████████░░░░░░░░  1s
Comment  ░░░░░░░░░░░░░░░░████████  1s
                            總耗時：3s
```

三個各 1 秒的 API，加總卻花了 3 秒。%( ´•̥̥̥ω•̥̥̥` )%

## 常見的解法

最直覺的解法，就是全在最外層用 `Promise.all` 一次打完：

```ts
const [user, postList, commentList] = await Promise.all([
  fetchUser(1),
  fetchPostList(1),
  fetchCommentList(1),
])
```

這雖然解決了瀑布流，卻會引發新問題：

- **資料與元件脫鉤**：原本各元件負責把自己的資料抓回來，現在全堆在父元件。
- **props 滿天飛**：子元件不能自己要資料，只能仰賴父元件層層把資料餵進來。
- **元件難以重用**：如果其他頁面想用同個元件，又得重新寫一遍抓資料的邏輯。

那有沒有辦法，既讓元件獨立抓自己的資料，又同時做到平行載入呢？

## Vue Data Loader

[Vue Data Loader](https://uvr.esm.is/data-loaders/) 是 `unplugin-vue-router` 提供的功能，核心精神是**將「資料請求」直接綁定在元件上，並交由路由在跳轉頁面前，統一提早觸發**。

這樣的設計，讓 Loader 能直接寫在**真正需要資料的子元件**裡，徹底保留了元件的獨立性。

我們可以直接在子元件（如 `PostCommentList.vue`）宣告並呼叫 Loader：

```vue
<!-- PostCommentList.vue -->
<script lang="ts">
import { defineBasicLoader } from 'unplugin-vue-router/data-loaders/basic'

// 1. 在子元件宣告 Loader。參數直接從路由 (to.params.id) 來，不用等父元件傳 props！
export const useCommentListLoader = defineBasicLoader('/posts/:id', async (to) => fetchCommentList(Number(to.params.id)))
</script>

<script setup lang="ts">
// 2. 拿資料囉！由於路由切換前早就提早打過 API，這裡的資料是「瞬間」備妥的！
const { data: commentList } = useCommentListLoader()
</script>

<template>
  <ul>
    <li v-for="comment in commentList" :key="comment.id">
      {{ comment.content }}
    </li>
    <post-related-list />
  </ul>
</template>
```

這種做法不但維持了「元件自管資料」的乾淨架構，多個 loader 甚至會在跳轉那瞬間**一起幫你平行抓好資料**：

```
Post     ████████░░░░░░░░░░░░░░░░  1s
Comment  ████████░░░░░░░░░░░░░░░░  1s
Related  ████████░░░░░░░░░░░░░░░░  1s
                            總耗時：1s
```

同樣三支 API，從 3 秒縮短到 1 秒。%( •̀ ω •́ )✧%

::: tip defineBasicLoader 為什麼要放在非 setup 的 `<script>` 中？
因為 loader 必須寫在**模組層級 (module level)**，路由器才有辦法在「切換頁面（元件還沒建立）」的階段，掃描到它並提前執行。

`<script setup>` 裡頭的程式碼屬於元件層級，一定要進到頁面、建立元件時才會跑。這就白白錯失了「換頁前預先平行抓取資料」的最佳時機。
:::

## Data Loader 如何處理資料依賴？

你可能會想：「要是 `PostList` 真的得靠 `User` 的 `id` 才能抓資料呢？不用 `v-if` 而改用 Data Loader 就能平行嗎？」

能不能平行，關鍵在於「依賴值從哪裡來」：

### 1. 依賴藏在 URL 裡（最常見的完美平行）

系統設計良好時，關鍵參數多半就寫在網址上（像 `/users/:id`）。
Loader 執行時會收到準備前往的路由物件 `to`，因此不論有幾個 Loader，全都可以**直接從網址解析出 `id`**，不必苦苦等別的 API 抓完再去抓。

```ts
// User Loader 與 Post Loader 都能直接過 to.params.id 取得 ID，達成平行發送！
export const useUserLoader = defineBasicLoader('/users/:id', async (to) => {
  return fetchUser(to.params.id)
})

export const usePostLoader = defineBasicLoader('/users/:id', async (to) => {
  return fetchPostList(to.params.id)
})
```

這麼一來，就算業務邏輯是「文章屬於使用者」，發送網路請求時兩邊依然能互不干擾、同時出發，把瀑布流殺得片甲不留。

### 2. 無法從 URL 取得的「真」依賴

有時依賴確實避不掉（譬如進 `/dashboard` 沒網址參數，一定得先打 API 取得 `userId = 123`，才能接續抓使用者的文章）。
這種情況物理上必定有先後順序、不可能同時跳過等待；但妙就妙在 Loader 發生於**切換頁面之前（導航階段）**，因此你能直接把這種串聯打法寫進 Loader 裡等待：

```ts
export const useDashboardLoader = defineBasicLoader('/dashboard', async () => {
  const user = await fetchMe()
  // 等待 fetchMe 回傳後再去抓文章
  const posts = await fetchPostList(user.id)
  return { user, posts }
})
```

雖然從網路看依舊是瀑布流，但優勢在於：**所有的等待，都被集中轉移到換頁前的「最頂層全局進度條」上了**。
等到畫面真正切換的那一刻，資料早就備齊。你永遠不必再看著畫面「轉圈圈消失 -> 彈出一半的樣式 -> 子元件又卡住開始轉另一個圈圈」，也順手消除了版面跳動（Layout Shift）的痛苦體驗！

## 總結 🐟

- 不管是不是用 `useAsyncState` 避開 `onMounted`，只要子元件被 `v-if` 擋住，就躲不掉瀑布流。
- 最外層全推給 `Promise.all` 確實能解決，卻把資料邏輯死綁在父元件上，犧牲了元件的獨立性。
- Vue Data Loader 讓載入動作跟著元件定義，再由路由器提早平行執行，效能跟程式碼乾淨度一次滿足。
- `defineBasicLoader` 要放在非 setup 的 `<script>` 才能提早被路由器偵測。
